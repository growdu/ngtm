# 逆天改命算命软件技术架构选型文档

## 1. 文档目标

本文档基于当前产品需求、概要设计、详细设计、UI 原型和模块设计，给出技术架构选型结论。

目标不是罗列可选技术，而是明确：

1. 当前阶段应该选什么
2. 为什么这样选
3. 放弃了哪些方案
4. 未来如何演进

---

## 2. 当前需求带来的技术约束

从现有文档看，系统有六个关键约束：

1. 产品核心是“持续交互演进画像”，不是一次性报告。
2. 需要同时处理结构化数据、半结构化证据和图片资源。
3. 存在大量异步计算链路，例如八字分析、图像分析、画像重算、人物匹配、建议生成、报告导出。
4. 需要强可解释性、强版本化和可回溯。
5. 多端目标明确，但 MVP 应优先快速验证核心闭环。
6. 后续可能需要更多规则、更多人物库、更多画像维度，架构必须可扩展。

因此，技术选型必须满足四点：

1. 开发效率高
2. 对结构化数据和 JSON 友好
3. 适合异步任务和版本化数据
4. 支持从 MVP 平滑演进到中型系统

---

## 3. 总体选型结论

推荐采用：

1. 前端 Web：`Next.js + TypeScript`
2. 移动端 Android / iOS：`React Native + Expo + TypeScript`
3. 桌面端：`Tauri`
4. 后端主服务：`FastAPI + Python`
5. 主数据库：`PostgreSQL + JSONB`
6. 缓存与队列：`Redis`
7. 异步任务：`Celery`
8. ORM 与迁移：`SQLAlchemy + Alembic`
9. 图片与文件存储：`S3 兼容对象存储`
10. 图像分析：`MediaPipe + OpenCV`
11. 报表与图形：`ECharts`
12. 部署方式：`Docker + 容器化部署`
13. API 形态：`REST 为主，SSE/轮询补充任务状态`
14. 跨端工程组织：`Monorepo + 共享 Schema / SDK / Design Tokens`

一句话概括：

当前最合适的是“Python 规则与计算中台 + Next.js Web + React Native(安卓/iOS) + PostgreSQL 版本化数据中心 + Redis/Celery 异步链路 + Monorepo 共享层”的架构。

---

## 4. 架构风格选型

## 4.1 选型结论

第一阶段采用：

`模块化单体 + 异步任务队列`

而不是一开始就做微服务。

## 4.2 选择理由

当前需求虽然模块多，但业务规模还远没有到必须拆微服务的程度。现在最重要的是验证闭环：

1. 基础信息录入
2. 初始分析
3. 问答校准
4. 画像版本演进
5. 历史人物匹配
6. 建议反馈回流

如果现在就拆成多个独立服务，会带来这些成本：

1. 联调成本高
2. 部署复杂度高
3. 链路追踪复杂
4. 早期迭代速度下降

所以第一阶段应采用：

1. 一个主后端仓库
2. 清晰模块边界
3. 通过 Celery 拆异步能力
4. 在代码结构上预留未来拆服务的边界

## 4.3 未来演进方式

当出现以下条件时再拆服务：

1. 图像分析耗时和资源消耗明显增大
2. 人物匹配需要独立扩容
3. 多团队并行开发导致模块耦合过高
4. 日活和计算量显著增长

优先拆出的服务应是：

1. 图像分析服务
2. 画像引擎服务
3. 历史人物匹配服务

## 5. 前端技术选型

## 5.1 跨平台候选方案总览

当前应正式比较三种跨平台路线：

1. `Next.js + React Native + Tauri`
2. `Flutter 全跨端`
3. `Tauri 2 全跨端`

### 方案 A：Next.js + React Native + Tauri

组成：

1. Web：`Next.js`
2. Android / iOS：`React Native + Expo`
3. Desktop：`Tauri`
4. 共享层：`TypeScript Monorepo`

特点：

1. Web 和移动端能力都成熟
2. 共享业务模型和接口成本低
3. 桌面端轻量
4. 不是单 UI 代码库，而是共享中间层

### 方案 B：Flutter 全跨端

组成：

1. Web：`Flutter Web`
2. Android / iOS：`Flutter`
3. Desktop：`Flutter Desktop`
4. 共享层：`Dart packages`

特点：

1. 真正意义上的单 UI 技术体系
2. Android / iOS 一致性最高
3. 桌面和移动统一性强
4. 与现有 React / TypeScript Web 生态割裂较大

### 方案 C：Tauri 2 全跨端

组成：

1. Web UI：前端框架自选
2. Android / iOS：`Tauri 2 Mobile`
3. Desktop：`Tauri 2`
4. Native bridge：`Rust + Swift/Kotlin`

特点：

1. 理论上最统一的壳层
2. 桌面端非常有优势
3. 移动端生态和工程成熟度仍弱于 React Native / Flutter
4. 更依赖 Rust 与原生桥接能力

## 5.2 方案对比表

| 维度 | Next.js + RN + Tauri | Flutter 全跨端 | Tauri 2 全跨端 |
| ---- | ---- | ---- | ---- |
| Web 能力 | 强 | 中 | 中到强 |
| Android / iOS 成熟度 | 强 | 强 | 中 |
| 桌面端成本 | 低 | 中 | 低 |
| 单代码库 UI 一致性 | 中 | 强 | 中 |
| 共享业务逻辑能力 | 强 | 强 | 中 |
| 相机/通知/相册能力 | 强 | 强 | 中 |
| SEO / 分享页能力 | 强 | 弱到中 | 取决于前端框架 |
| 团队上手门槛 | 中 | 中 | 高 |
| Rust 依赖程度 | 低 | 无 | 高 |
| 当前产品匹配度 | 强 | 中到强 | 中 |

## 5.3 当前产品视角下的判断

你的产品不是纯内容站，也不是纯移动工具，而是同时具备：

1. Web 端深度查看和时间线浏览
2. Android / iOS 端拍照、提醒、碎片化问答
3. 桌面端长文本录入、文件导出和外设接入

所以不能只问“哪种跨平台最统一”，而要问：

1. 哪种方案最适合 Web
2. 哪种方案最适合 Android / iOS
3. 哪种方案能控制整体工程复杂度

按这个标准，三案判断如下：

### 方案 A 判断

最平衡，工程风险最低，最适合当前阶段。

### 方案 B 判断

如果你决定“移动优先，并愿意牺牲一部分 Web 生态协同”，Flutter 可以成为强竞争方案。

### 方案 C 判断

更适合 Rust 能力强、追求统一壳层、且能接受移动端桥接成本的团队，不适合作为当前默认方案。

## 5.4 推荐结论

当前推荐顺序：

1. 第一推荐：`Next.js + React Native + Tauri`
2. 第二推荐：`Flutter 全跨端`
3. 第三推荐：`Tauri 2 全跨端`

推荐原因：

1. 你的产品 Web 入口价值高，不能弱化 Web。
2. Android / iOS 需要成熟的拍照、通知、表单和高频交互能力。
3. 桌面端存在但不是第一优先级，Tauri 足够合适。
4. 这套组合在“开发效率、成熟度、跨端平衡、后续扩展”之间最稳。

## 5.5 什么时候应改选 Flutter

满足以下条件时，可把 Flutter 升为第一推荐：

1. 你明确要求 Android 和 iOS 成为最核心平台
2. Web 只需要中等强度展示，不强调 SEO / 分享页 / React 生态
3. 团队接受 Dart 作为核心语言
4. 你更重视单一 UI 技术体系，而不是 Web/移动分层最优

## 5.6 什么时候应改选 Tauri 2 全跨端

满足以下条件时，可认真评估：

1. 团队具备较强 Rust 能力
2. 桌面端和移动端都希望统一壳层
3. 接受 Android / iOS 部分能力通过原生桥或插件补齐
4. 可以承担较高的工程验证成本

## 5.7 跨平台前端总体策略

### 结论

前端采用“多端独立壳 + 共享中间层”策略：

1. Web：`Next.js`
2. Android / iOS：`React Native + Expo`
3. Desktop：`Tauri`
4. 共享层：`TypeScript packages`

### 共享内容

跨平台统一共享以下内容：

1. API 类型定义
2. 请求 SDK
3. 画像、匹配、建议等领域模型
4. 表单校验 Schema
5. 设计 Token
6. 状态枚举与常量

### 不共享内容

以下部分按端独立实现：

1. 页面路由
2. 原生拍照、相册、通知能力
3. 桌面端文件与窗口能力
4. Web SEO 和分享页

### 选型原则

跨平台不等于所有 UI 一套代码跑所有端。更合理的是：

1. 业务能力统一
2. API 统一
3. 类型统一
4. 设计系统统一
5. 端侧渲染分别实现

这样既能兼顾 Android/iOS，也不会牺牲 Web 和桌面端体验。

## 5.8 Web 端：Next.js + TypeScript

### 结论

Web 端优先采用 `Next.js`。

### 原因

1. 当前最需要先做 MVP，Web 端验证速度最快。
2. `Next.js` 适合做内容展示、复杂表单、时间线、图表、版本对比、问答流程。
3. 服务端渲染和路由能力成熟，便于后续做 SEO、落地页和分享页。
4. TypeScript 对复杂页面状态、组件接口和视图模型更稳。

### 适配本项目的点

1. 首页、画像页、人物匹配页、成长档案页都更适合 Web 首发。
2. 管理复杂交互比纯移动端更高效。
3. 便于做 PDF 导出前的 HTML 模板预览。

### 推荐配套

1. UI：`Tailwind CSS`
2. 图表：`ECharts`
3. 表单：`React Hook Form + Zod`
4. 数据请求：`TanStack Query`
5. 状态：局部状态为主，全局只保留少量会话状态

## 5.9 为什么不是纯 React + Vite

`React + Vite` 也可行，但当前有两点让 `Next.js` 更合适：

1. 后续分享页、导出页、营销页更适合 Next.js
2. 页面路由、服务端渲染和 BFF 协作更顺

如果项目永远只做内部工具型前端，Vite 更轻；但当前产品不是这个方向。

## 5.10 Android / iOS：React Native + Expo

### 结论

Android 和 iOS 建议采用 `React Native + Expo`，并从第一阶段就纳入总体架构设计。

### 原因

1. 当前产品天然需要移动端能力：拍照上传、通知提醒、事件记录、碎片化问答。
2. React 技术栈可以与 Web 共享类型、接口定义、业务模型和部分状态逻辑。
3. Expo 能显著降低 Android/iOS 双端工程门槛。
4. 对相机、相册、推送、文件访问等能力有成熟支持。

### 适配本项目的点

1. 面相/手相拍照采集更适合手机端。
2. 吉日提醒、建议打卡、成长反馈更适合移动端高频触达。
3. 持续问答在移动端通常比桌面端转化更高。

### 推荐配套

1. `Expo Router`
2. `TanStack Query`
3. `Zod`
4. `NativeWind` 或统一 Token 映射方案
5. `expo-notifications`
6. `expo-image-picker` / `expo-camera`

### 实施建议

1. 即使 MVP 首发以 Web 为主，后端 API、鉴权、上传、任务状态、通知模型也要按 Android/iOS 一并设计。
2. 所有与端无关的 DTO、schema、状态枚举都放共享包。

## 5.11 为什么不是 Flutter 作为默认首选

Flutter 也能做，但当前不优先：

1. 当前已经明确 Web 是重要入口，React 体系在 Web 与移动之间复用率更高。
2. 后续共享类型、API SDK、设计 Token 时，TypeScript 方案更顺。
3. 当前产品更强调数据化界面、表单、图表、问答流，而不是重自绘渲染。

如果未来战略转为“移动优先且几乎不重视 Web”，Flutter 可以重新评估，甚至可能变成第一选择，但不是当前默认最优。

## 5.12 桌面端：Tauri

### 结论

桌面端建议采用 `Tauri`。

### 原因

1. 桌面端核心需求是长文本录入、通知、文件导出、摄像头接入。
2. Tauri 更轻量，资源占用小于 Electron。
3. 如果主前端是 Web 技术栈，迁移成本低。

## 5.13 为什么不是 Electron

Electron 的生态更成熟，但本项目当前不优先：

1. 安装包更大
2. 内存占用更高
3. 现阶段桌面端不是主战场，不值得先承担这个成本

除非后续需要大量 Node 生态桌面能力，否则 Tauri 更合理。

## 5.14 跨端工程组织：Monorepo

### 结论

推荐采用 `Monorepo` 组织 Web、Android、iOS、Desktop 与共享包。

### 推荐结构

```text
apps/
  web/         -> Next.js
  mobile/      -> React Native + Expo
  desktop/     -> Tauri shell
services/
  api/         -> FastAPI
  worker/      -> Celery workers
packages/
  api-schema/  -> OpenAPI types / DTO
  sdk/         -> API client
  domain/      -> profile/match/advice shared models
  validation/  -> Zod schemas
  design-tokens/
```

### 原因

1. Android/iOS/Web/Desktop 可以共享模型和约束。
2. 接口变更一次，多端同步更新。
3. 避免多仓库带来的契约漂移。

## 6. 后端技术选型

## 6.1 主后端：FastAPI + Python

### 结论

后端核心服务采用 `FastAPI`。

### 原因

1. 本项目大量规则计算、文本解释、图像处理和异步任务都天然适合 Python。
2. `FastAPI` 在接口定义、类型约束、Pydantic 校验和开发效率上很适合中台型系统。
3. 与 `MediaPipe`、`OpenCV`、数据分析库和 LLM 接入都更自然。

### 适配本项目的能力

1. REST API 输出清晰
2. 后续可补充后台管理 API
3. 与 Celery、SQLAlchemy、Alembic 组合成熟

## 6.2 为什么不是 Django

Django 也可以做，但不是当前最优：

1. 本项目不是典型 CMS 或管理后台先行项目
2. 核心在计算链路与异步编排，`FastAPI` 更轻
3. Pydantic 风格与复杂 API 契约更贴合

如果未来需要重管理后台，可额外加管理台，而不必用 Django 统治全项目。

## 6.3 API 形态：REST 为主

### 结论

主 API 使用 `REST`，不建议首阶段采用 GraphQL。

### 原因

1. 模块边界清晰，REST 更直观。
2. 画像、匹配、建议、问答、版本等资源天然适合资源化接口。
3. 异步任务较多，前端更需要“状态查询”而不是自由拼装字段。
4. GraphQL 在早期会增加 schema 维护和聚合复杂度。

### 补充机制

1. 长任务状态用轮询或 SSE
2. 页面聚合由 BFF 完成

## 7. 数据存储选型

## 7.1 主数据库：PostgreSQL + JSONB

### 结论

主数据库采用 `PostgreSQL`，大量动态特征字段使用 `JSONB`。

### 原因

1. 画像版本、匹配结果、建议结果既有固定字段，也有动态扩展字段。
2. `JSONB` 很适合存 trait、evidence、confidence、difference breakdown 等结构。
3. PostgreSQL 对事务、一致性、复杂查询、索引、时间线查询都足够强。
4. 非常适合做版本化审计型业务系统。

### 本项目最适合 PostgreSQL 的点

1. 画像版本快照
2. 证据链回溯
3. 人生事件时间线
4. 历史人物结构化 traits
5. 建议执行反馈

## 7.2 为什么不是 MongoDB

MongoDB 在文档存储上看似方便，但当前不推荐：

1. 本项目核心是强关联、强版本、强审计
2. 事务与复杂关系查询不如 PostgreSQL 稳
3. 后续画像版本对比、事件时间线、建议回流都更适合关系型数据库

## 7.3 Redis

### 结论

采用 `Redis` 做缓存、去重、分布式锁和 Celery Broker。

### 用途

1. 任务队列
2. 热点画像缓存
3. 幂等 key
4. 短期会话状态
5. 通知去重

## 7.4 对象存储

### 结论

图片、报告、分享文件使用 `S3 兼容对象存储`。

### 推荐

1. 开发环境：`MinIO`
2. 生产环境：云厂商对象存储

### 原因

1. 面相、手相图片不能落本地磁盘
2. 报告导出和分享海报也适合对象存储
3. 便于后续 CDN 加速

## 8. 异步任务与计算链路选型

## 8.1 Celery

### 结论

异步任务框架采用 `Celery`。

### 原因

1. Python 生态成熟
2. 适合多阶段任务链
3. 支持重试、任务路由、定时任务和状态管理

### 本项目适用任务

1. 八字分析
2. 面相分析
3. 手相分析
4. 画像重算
5. 人物匹配重算
6. 建议生成
7. 报告导出
8. 通知发送

## 8.2 为什么不是 RQ

RQ 更轻，但当前需求不只是简单后台 job：

1. 任务种类较多
2. 需要链式触发
3. 需要更完整的重试和编排能力

因此 Celery 更稳。

## 8.3 任务流建议

推荐任务链：

```text
基础信息提交
-> analyze_bazi
-> recompute_profile
-> recompute_match
-> generate_advice

图片上传
-> analyze_face / analyze_palm
-> recompute_profile
-> recompute_match
-> generate_advice

问答/事件/反馈提交
-> recompute_profile
-> recompute_match
-> generate_advice
-> send_notification
```

## 9. 图像处理技术选型

## 9.1 MediaPipe + OpenCV

### 结论

图像分析采用 `MediaPipe + OpenCV`。

### 分工建议

1. `MediaPipe` 负责关键点检测
2. `OpenCV` 负责预处理、裁剪、增强、质量评估和辅助特征提取

### 原因

1. 当前需求更偏“结构特征提取”，而不是训练自定义大模型
2. 先用现成稳定工具更适合 MVP
3. 面部关键点、手部关键点这类任务，MediaPipe 成熟度足够

## 9.2 为什么不先做自训练模型

当前不应一开始投入自训练模型，原因很直接：

1. 数据集没有准备好
2. 需求尚未稳定
3. 规则链路和画像逻辑才是当前瓶颈
4. 过早训练会拉高成本和项目风险

更合理的顺序是：

1. 先跑通关键点提取 + 规则映射
2. 再评估识别准确率
3. 最后决定是否需要定制模型

## 10. 画像与匹配计算架构选型

## 10.1 规则驱动优先

### 结论

画像引擎和人物匹配引擎优先采用：

`规则驱动 + 结构化加权计算`

而不是首阶段依赖黑盒模型。

### 原因

1. 当前产品强调可解释性
2. 用户需要看到“为什么变了”
3. 历史人物匹配需要维度拆解，而不是一句模糊判断
4. 规则算法更适合版本化和审计

## 10.2 LLM 的角色

LLM 只负责：

1. 润色解释文案
2. 总结变化说明
3. 将结构化结果转为可读文本

LLM 不负责：

1. 最终 trait 分值
2. 最终匹配排序
3. 关键建议优先级

## 10.3 是否引入向量检索

### 结论

首阶段不强依赖向量数据库。

### 方案

1. 历史人物匹配先用 PostgreSQL 存结构化 traits
2. 采用加权距离/相似度计算
3. 如果后续人物库非常大，再评估 `pgvector`

### 原因

当前人物库规模不会大到必须独立引入向量引擎。

## 11.2 为什么不先上 WebSocket

这类需求用轮询或 SSE 足够，复杂度低于 WebSocket。

## 18. 最终决策摘要

当前需求下，最合理的技术架构不是“大而全”，而是：

1. 以 `Next.js + React Native + FastAPI + PostgreSQL + Redis + Celery` 作为跨平台主干
2. 从第一天就把 Android/iOS 纳入统一 API、统一鉴权、统一上传、统一通知模型
3. 先把“证据 -> 画像版本 -> 人物匹配 -> 建议回流”做稳定
4. 先用规则和结构化计算保证可解释性
5. 把高复杂度图像分析和独立服务拆分放到第二阶段以后

如果只保留一句结论：

本项目当前最优选型是“跨平台优先、Web 与 Android/iOS 同构设计、Python 中台、PostgreSQL 为核心、异步任务驱动、规则引擎优先、Monorepo 共享层驱动多端演进”的架构。
