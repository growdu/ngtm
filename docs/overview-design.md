# 逆天改命算命软件概要设计文档

## 1. 文档目的

本文档用于说明“逆天改命算命软件”的总体技术方案、系统边界、核心模块划分、关键数据流和部署思路，为后续详细设计、接口开发和前后端协作提供统一基础。

---

## 2. 设计目标

本系统不是一次性出结论的算命工具，而是一个持续交互演进的用户画像系统。

系统目标：

1. 基于用户基础资料、图像资料、行为问答、人生事件构建初始画像。
2. 随着用户持续提供新信息，动态修正画像与结论。
3. 对每次画像更新提供可解释的变化说明与置信度。
4. 基于画像匹配历史人物，并给出相似点、差异点和成长建议。
5. 将命理分析、图像分析、行为校准、建议追踪统一到一套可版本化的中台能力中。

---

## 3. 系统范围

### 3.1 业务范围

系统包含以下核心业务：

1. 用户基础信息采集
2. 八字分析
3. 面相分析
4. 手相分析
5. 多轮问答与人生事件采集
6. 动态用户画像生成与版本演进
7. 历史人物匹配
8. 逆天改命建议生成与执行追踪
9. 历史记录、版本对比、报告导出与分享

### 3.2 非目标范围

当前阶段不纳入以下目标：

1. 复杂社交关系网络
2. 实时多人协同
3. 自定义规则编辑后台
4. 全自动训练型大模型平台

---

## 4. 总体架构

系统采用前后端分离、领域服务拆分、规则引擎与生成式文案分离的架构。

### 4.1 架构分层

1. 表现层
   Web、移动端、桌面端，负责信息录入、图像采集、结果展示、持续问答、时间线和报告导出。
2. 接入层
   API Gateway/BFF，负责鉴权、限流、会话管理、端能力适配。
3. 业务服务层
   用户服务、采集服务、八字服务、图像分析服务、画像服务、人物匹配服务、建议服务、报告服务。
4. 计算与规则层
   八字规则库、图像特征规则、画像融合引擎、匹配引擎、建议规则引擎、LLM 文案生成器。
5. 数据层
   PostgreSQL、对象存储、缓存、异步任务队列、日志与审计存储。

### 4.2 逻辑架构图

```text
Client(Web/Mobile/Desktop)
        |
   API Gateway / BFF
        |
 ---------------------------------------------------------
 | User Service | Intake Service | Profile Service        |
 | Bazi Service | Vision Service | Match Service          |
 | Advice Service | Report Service | Notification Service |
 ---------------------------------------------------------
        |
 ---------------------------------------------------------
 | Rule Engine | Profile Fusion Engine | Match Engine     |
 | LLM Adapter | Scheduler | Async Worker               |
 ---------------------------------------------------------
        |
 -------------------------------
 | PostgreSQL | Object Storage |
 | Redis      | Queue          |
 -------------------------------
```

## 5. 核心设计原则

### 5.1 持续演进优先

所有核心结论必须支持后续修正。系统不把第一次分析视为最终答案，而视为第一版假设。

### 5.2 可解释优先

任何核心输出都应能回答三个问题：

1. 结论来自哪些输入
2. 当前结论的置信度是多少
3. 相比上一版为何发生变化

### 5.3 规则与文案分离

评分、画像、匹配、排序由规则与算法负责；自然语言解释、报告润色由生成式模型负责，避免系统核心逻辑不可控。

### 5.4 版本化优先

用户画像、分析结果、匹配结果、建议结果都必须带版本号，并可回溯。

### 5.5 异步计算优先

图像分析、画像重算、人物匹配、PDF 导出等耗时任务应使用异步任务处理，避免阻塞主交互流程。

## 6. 核心模块设计

### 6.1 用户与会话模块

职责：

1. 用户身份识别
2. 多端会话同步
3. 当前画像版本定位
4. 用户分析进度管理

### 6.2 信息采集模块

职责：

1. 基础信息录入
2. 图像资料上传
3. 多轮问答
4. 人生关键事件录入
5. 用户反馈采集

特点：

1. 支持分阶段采集
2. 支持跳过与补录
3. 支持识别“高价值缺失信息”

### 6.3 八字分析模块

职责：

1. 四柱计算
2. 五行统计
3. 十神分析
4. 大运流年分析
5. 命理解释结构化输出

特点：

1. 原始命盘固定
2. 命盘解释支持结合用户反馈动态调整

### 6.4 图像分析模块

职责：

1. 面相关键点识别
2. 手相线条与手型识别
3. 图像质量评估
4. 结构化特征输出

特点：

1. 多张图片可交叉校验
2. 低质量图片不直接进入高置信度结论

### 6.5 画像引擎模块

职责：

1. 汇总多源输入
2. 融合证据并计算画像维度
3. 输出画像版本
4. 记录结论变化

这是系统中台核心。

### 6.6 历史人物匹配模块

职责：

1. 历史人物画像库维护
2. 用户画像向量化或结构化映射
3. 匹配排序
4. 相似与差异解释生成

### 6.7 建议与追踪模块

职责：

1. 生成个性化建议
2. 拆解为可执行任务
3. 追踪执行反馈
4. 根据执行结果反哺画像

### 6.8 报告与展示模块

职责：

1. 动态画像展示
2. 历史人物结果页
3. 时间线与版本对比
4. PDF 导出与分享

## 7. 关键业务流程

### 7.1 初始建档流程

```text
用户填写基础信息
-> 系统生成初始八字结果
-> 用户上传面相/手相图片
-> 系统形成第一版画像
-> 系统识别关键缺失项
-> 发起下一轮问答
```

### 7.2 持续演进流程

```text
用户补充问答/经历/反馈
-> 采集服务写入新证据
-> 触发画像重算任务
-> 生成新画像版本
-> 重跑历史人物匹配
-> 生成变化说明与新建议
-> 通知用户查看更新
```

### 7.3 建议追踪闭环

```text
系统生成建议
-> 用户选择执行
-> 系统设定提醒与跟踪点
-> 用户回填执行效果
-> 系统判断建议有效性
-> 更新画像与建议优先级
```

## 8. 数据设计概览

核心数据实体：

1. User
2. IntakeRecord
3. BaziAnalysis
4. FaceAnalysis
5. PalmAnalysis
6. LifeEvent
7. UserProfileVersion
8. HistoricalFigure
9. MatchResult
10. AdvicePlan
11. AdviceExecution

数据存储策略：

1. 结构化字段落 PostgreSQL 表字段
2. 可扩展维度使用 JSONB
3. 图片文件落对象存储
4. 任务与缓存信息落 Redis/Queue
5. 审计与推理日志单独保留

## 9. 接口设计概览

接口分为五类：

1. 采集类接口
2. 分析类接口
3. 画像类接口
4. 匹配类接口
5. 建议与报告类接口

典型接口：

1. `POST /api/v1/intake/basic`
2. `POST /api/v1/intake/images`
3. `POST /api/v1/intake/questionnaire/answer`
4. `POST /api/v1/events`
5. `POST /api/v1/profile/recompute`
6. `GET /api/v1/profile/current`
7. `GET /api/v1/profile/versions`
8. `GET /api/v1/match/current`
9. `GET /api/v1/advice/current`
10. `POST /api/v1/advice/execution`

## 10. 部署设计

### 10.1 部署拓扑

1. 前端静态资源部署到 CDN
2. 网关与业务服务部署到容器平台
3. PostgreSQL 作为主数据库
4. Redis 用于缓存和任务调度
5. 对象存储保存图片与导出文件
6. 后台 Worker 处理异步计算

### 10.2 部署形态

建议分三套环境：

1. 开发环境
2. 测试环境
3. 生产环境

### 10.3 扩展策略

1. 图像分析服务可独立扩容
2. 画像引擎和匹配引擎可拆为独立计算服务
3. 报告导出与通知服务可单独扩容

---

## 11. 安全与合规

### 11.1 数据安全

1. 图片与个人信息需访问控制
2. 敏感字段脱敏展示
3. 用户删除与数据导出机制

### 11.2 可审计性

1. 保存关键画像更新日志
2. 保存关键建议生成来源
3. 保存匹配结果计算版本

### 11.3 风险提示

1. 产品应明确提示结果仅供参考
2. 对健康、寿命、财务等高风险表述进行合规降级
3. 不输出极端确定性结论

---

## 12. 非功能需求

### 12.1 性能

1. 基础信息提交接口响应时间小于 500ms
2. 画像重算采用异步，普通任务在 10 秒内完成
3. 图像分析任务在 30 秒内完成

### 12.2 可用性

1. 核心 API 可用性目标 99.9%
2. 失败任务支持重试
3. 画像版本生成必须幂等

### 12.3 可维护性

1. 规则库配置化
2. 画像维度与匹配维度可扩展
3. 任务与接口全链路日志可追踪

---

## 13. MVP 范围建议

优先实现以下闭环：

1. 基础信息录入
2. 八字分析
3. 多轮问答
4. 第一版画像生成
5. 画像版本更新
6. Top 3 历史人物匹配
7. 基础建议生成

暂缓内容：

1. 高精度面相手相识别
2. 复杂建议追踪体系
3. 多端原生高级能力

---

## 14. 技术选型建议

1. 后端：FastAPI
2. 数据库：PostgreSQL + JSONB
3. 缓存与队列：Redis
4. 异步任务：Celery/RQ
5. Web 前端：Next.js
6. 图像分析：MediaPipe + OpenCV
7. 报告导出：HTML 模板转 PDF
